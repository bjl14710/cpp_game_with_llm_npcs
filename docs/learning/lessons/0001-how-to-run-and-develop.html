The two environments

This project runs in two completely separate environments and you use them for different jobs. Mixing them up is the #1 source of "why doesn't this work?" moments.

┌─────────────────────────────┐ ┌─────────────────────────────┐ │ Linux / Dev Container │ │ Windows (your machine) │ │ │ │ │ │ g++, make, git, curl │ │ MSYS2 UCRT64 + SFML │ │ NO cmake, NO SFML │ │ Ollama (model loaded) │ │ │ │ │ │ → make -C tests test │ │ → run.bat (double-click) │ │ (fast, offline) │ │ (builds + plays) │ └─────────────────────────────┘ └─────────────────────────────┘ primary dev loop visual + play verification

Container / Linux is where you write code and run the unit tests. All 57 tests run in under two seconds, fully offline — no Ollama, no SFML, no window. This is your main feedback loop.

Windows is where the game actually renders. The 3D city, the NPC figures, the dialogue overlay — none of that can be verified without SFML and OpenGL. You push your commits, then pull and run run.bat on Windows to see how it looks.

Key rule
Logic in src/core/ is tested in the container. Visual behaviour in src/app/ is verified on Windows. Never reverse this.

Running the game on Windows

All prerequisites must be in place first. Here is the full one-time setup and the daily run flow.

One-time setup

  1. Install Ollama from https://ollama.com/download, then in any terminal:
    ollama pull qwen2.5:3b-instruct
    This pulls a ~2 GB model. It only happens once (or when you want to try a different model).
  2. Install MSYS2 from https://www.msys2.org. Open the UCRT64 shell (not the default MSYS shell — ucrt64 specifically) and run: 
    pacman -S mingw-w64-ucrt-x86_64-gcc \
           mingw-w64-ucrt-x86_64-cmake \
           mingw-w64-ucrt-x86_64-sfml
    This gives you the compiler, build system, and graphics library the game needs.
  3. Clone the repo (if you haven't): 
    git clone https://github.com/bjl14710/cpp_game_with_llm_npcs.git
cd cpp_game_with_llm_npcs
git checkout initial_npcs_and_world

Every time you want to play

  1. Make sure Ollama is running: ollama serve in a terminal (or it may already be running as a service).
  2. Double-click run.bat in the repo folder. That's it.

What run.bat actually does, step by step:

REM 1. Sets MSYS2 ucrt64 tools on PATH so cmake and g++ can be found
set "PATH=C:\msys64\ucrt64\bin;%PATH%"

REM 2. Checks Ollama is reachable — if not, prints a helpful error and pauses
curl -fsS --max-time 2 "http://localhost:11434/api/tags"

REM 3. Configures the build (only matters the first time or after CMakeLists changes)
cmake -S . -B build -G "MinGW Makefiles"

REM 4. Compiles everything (parallel, fast on second run since CMake is incremental)
cmake --build build -j

REM 5. Runs the game
build\cpp_game_with_llm_npcs.exe
Common issue
If you get "MSYS2 ucrt64 toolchain not found", the MSYS2 install is at a non-default path or you installed the wrong variant. run.bat looks for C:\msys64\ucrt64\bin — check that path exists.

The development loop (day-to-day)

You will spend most of your time in the container (Linux), not on Windows. Here is the cycle:

  1. Edit code in src/core/ or tests/
  2. Run tests — one command, takes ~1 second: 
    make -C tests test
    You'll see each compiled file scroll by, then a pass/fail summary from doctest.
  3. Fix any failures, then repeat steps 1–2 until green.
  4. Commit and push — small commits (2–3 files): 
    git add src/core/Npc.hpp src/core/Npc.cpp
git commit -m "feat(core): add patrol state to Npc"
git push origin initial_npcs_and_world
  5. On Windows: git pull, then run.bat to see it rendered.

What the test command actually runs

The command make -C tests test is shorthand for "run the Makefile inside the tests/ directory, targeting the test rule." That Makefile does:

  1. Compiles every src/core/*.cpp file
  2. Compiles every tests/test_*.cpp file
  3. Links them all into tests/build/run_tests
  4. Executes that binary — doctest reports results

Notice it does not compile anything in src/app/. That is intentional — the app layer needs SFML which is not in the container.

Current state
57 tests, 402 assertions — all pass in this repo as of lesson creation. If you add new logic to src/core/, add tests in the same commit.

The core / app split — why it's designed this way

The repo's source tree is divided into two halves with a hard rule between them:

FolderContainsAllowed to useTested by
src/core/All game logicstdlib, httplib, json, pthreadsdoctest in container
src/app/Graphics + inputSFML, OpenGL, plus coreManual on Windows

The rule: nothing in src/core/ may include an SFML or OpenGL header. Ever.

Why does this matter? Because SFML cannot be installed in the container. If City.cpp included <SFML/Graphics.hpp>, the test suite would fail to compile and you'd have no automated safety net. The split buys you:

  • Fast offline tests — all logic is reachable from the container, no GUI needed.
  • Portability — the core could be ported to a mobile game, a web server, a CLI tool. The app layer is the only part that is Windows-locked.
  • Confidence before you see it — when NPC collision or dialogue logic is green in tests, you know it's correct before you ever run the game.

The CMake structure mirrors this split

# llm_npc_core: always builds, no SFML
add_library(llm_npc_core STATIC
    src/core/City.cpp    src/core/Config.cpp
    src/core/LlmClient.cpp  src/core/Npc.cpp  # ... etc
)

# Game executable: only built where SFML + OpenGL exist
if(SFML_FOUND AND OPENGL_FOUND)
    add_executable(cpp_game_with_llm_npcs
        src/app/main.cpp  src/app/Renderer3D.cpp  # ...
    )
    target_link_libraries(...  llm_npc_core  sfml-graphics ...)
endif()

The container runs CMake and gets only the library. Windows gets both the library and the executable. The tests always link against llm_npc_core directly, bypassing the exe entirely.

Often confused with: make vs cmake

This project uses both, in different places. They are not the same thing.

makecmake
What it is A build runner that reads a Makefile you write by hand A build-system generator — it produces a Makefile (or VS project, or Ninja files) from a CMakeLists.txt
In this project tests/Makefile — hand-written, directly drives g++ for the test suite in the container CMakeLists.txt — used on Windows to produce the MinGW Makefile that builds the full game
When you run it make -C tests test (container, fast, no install needed) cmake -S . -B build -G "MinGW Makefiles" (Windows, once — then cmake --build build)
Why two? The container has make but not cmake. The hand-written Makefile gets tests running without requiring an extra install. cmake is only needed on Windows where it handles finding SFML/OpenGL for you.

Rule of thumb: in the container, use make -C tests test. On Windows, use run.bat (which calls cmake internally).

Practice — check your understanding

Options are shuffled on load. Click an answer to see if you're right and why.

1. You edit src/core/City.cpp to change how buildings are sized. What is the correct next step?

  • Run make -C tests test in the container
  • Double-click run.bat on Windows
  • This is an app-only change; only verify it on Windows
  • Commit and push immediately without running tests

2. A teammate adds a new module src/core/QuestLog.hpp that includes <SFML/Graphics.hpp>. What is wrong?

  • It violates the core/app split — core must never include SFML
  • Nothing — SFML includes are fine anywhere in the project
  • Quest log logic belongs in src/app/ regardless of includes
  • CMake will auto-detect and fix this at configure time

3. run.bat says "Ollama is not reachable at localhost:11434". What are the two most likely causes?

  • Ollama is not running / ollama serve was never started
  • The model qwen2.5:3b-instruct was not pulled
  • MSYS2 is not installed
  • cmake configuration failed

4. What is the difference between make and cmake in this project?

  • make runs the hand-written tests/Makefile in the container; cmake generates the Windows build files from CMakeLists.txt
  • make is for Windows; cmake is for Linux
  • cmake runs the tests; make builds the release executable
  • They are interchangeable — use whichever you prefer on either platform

5. You want to add an 11th NPC. Which files need to change, and do you need to rebuild run.bat?

  • Create a new personas/<id>.persona file — no code changes or build changes needed
  • Edit src/core/World.cpp to increase the hardcoded NPC limit
  • Update both run.bat and CMakeLists.txt
  • Only edit src/app/Renderer3D.cpp — no .persona file needed
